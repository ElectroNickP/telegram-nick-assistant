# Telegram Nick Assistant

Минимальный проект: userbot (Telethon) получает все входящие сообщения во всех чатах; при ответе на ваше сообщение или при упоминании (@username / по ID) отдельный бот отправляет уведомление в указанный чат с кнопкой «Перейти к сообщению».

## Запуск

1. Установите зависимости:

   ```bash
   pip install -r requirements.txt
   ```

2. Скопируйте `.env.example` в `.env` и заполните переменные:

   ```bash
   cp .env.example .env
   ```

3. Запустите:

   ```bash
   python main.py
   ```

При первом запуске Telethon запросит номер телефона и код подтверждения; после этого сессия сохраняется в файл (имя задаётся в `SESSION_NAME`) и повторный ввод не нужен.

## Переменные окружения

| Переменная | Описание | Где взять |
|------------|----------|-----------|
| `API_ID` | ID приложения Telegram | [my.telegram.org](https://my.telegram.org) — создать приложение |
| `API_HASH` | Hash приложения Telegram | Там же |
| `BOT_TOKEN` | Токен бота для уведомлений | [@BotFather](https://t.me/BotFather) — создать бота |
| `NOTIFICATION_CHAT_ID` | Куда слать уведомления (обычно ваш user ID = личка с ботом) | Например [@userinfobot](https://t.me/userinfobot) |
| `SESSION_NAME` | Имя файла сессии userbot (без расширения) | Например `nick_assistant_session` |
| `LOG_LEVEL` | Уровень логирования (по умолчанию INFO) | Опционально |

Сессия userbot хранится в корне проекта в файле `{SESSION_NAME}.session`; этот файл не должен попадать в репозиторий (указан в `.gitignore`).

## Зависимости

- Python 3.10+
- telethon
- aiohttp
- python-dotenv

Без БД, веб-сервера и AI: один процесс, один entry point `main.py`.

## Почему бот не реагирует на некоторые сообщения

- **Сообщения от вас самих** — уведомления не отправляются, когда вы сами пишете в чат (в т.ч. если вы написали @ваш_username). Так и задумано: вы не получаете уведомление о своём же сообщении.
- **Бот был выключен** — уведомления приходят только за то время, когда процесс `main.py` (или сервис) был запущен. Сообщения, пришедшие до старта или во время простоя, не обрабатываются.
- **Только входящие** — учитываются только входящие сообщения (не ваши исходящие). Ответы на ваши сообщения и упоминания @username или по ID — обрабатываются.

**Как понять, почему сообщение пропущено:** смотрите логи (`journalctl --user -u telegram-nick-assistant -n 50` или вывод `main.py`). В логах будет:
- `Skip: сообщение от вас` — вы сами написали (уведомления о своих сообщениях не шлём);
- `Skip: сообщение без текста` — пришло только медиа/стикер;
- `Notification sent` — уведомление отправлено.

Для деталей по упоминаниям (какой @ не совпал и т.п.) задайте в `.env` `LOG_LEVEL=DEBUG` и перезапустите проект.

## Автозапуск и трей

Чтобы проект автоматически запускался при входе в систему и перезапускался при сбоях, а в трее всегда был значок для запуска/остановки:

1. Установите зависимости трея (один раз):

   ```bash
   pip install -r requirements-tray.txt
   ```
   или после создания venv: `./.venv/bin/pip install -r requirements-tray.txt`

2. Запустите установщик (без sudo — используется systemd пользователя):

   ```bash
   chmod +x install-tray-and-service.sh
   ./install-tray-and-service.sh
   ```

Перед установкой закройте любой вручную запущенный `main.py`, чтобы не было блокировки сессии (один процесс на одну сессию).

После этого:
- **Сервис** (systemd --user) стартует при входе в сессию и перезапускается при падении (Restart=on-failure).
- **Трей** добавляется в автозапуск (~/.config/autostart); при входе появляется значок (зелёный = работает, серый = остановлен).
- В меню трея: **Start**, **Stop**, **Restart**, **Control** (окно с кнопками), **Quit** (только закрыть трей, сервис продолжает работать).

Удаление: `./uninstall-tray-and-service.sh`

## Конфликт с Best My Assistant (два userbot'а на один аккаунт)

**Симптомы:** оба проекта постоянно перезагружаются (restart counter растёт у Nick и у Best). В логах Nick — TypeNotFoundError 94345242; в логах Best — TypeNotFoundError 9815cec8. В «Remaining bytes» у Nick виден текст от Best («Система запущена!», «Best-ElectroNick» и т.д.); у Best — текст от Nick («Telegram Nick Assistant запущен...»).

**Причина:** один и тот же Telegram-аккаунт (Nick @Pankonick) используется **двумя** userbot'ами **одновременно** (Best My Assistant и Telegram Nick Assistant). Они работают с разными файлами сессии, но с одним аккаунтом. При вызове getDifference() сервер отдаёт обновления для аккаунта; в поток попадают TL-типы, порождённые другим клиентом. Telethon каждого приложения не знает «чужой» тип → падение → перезапуск. Подробно: `оперативка/РАССЛЕДОВАНИЕ_ПЕРЕЗАГРУЗОК_31_01_2026.md`.

**Решение (обязательно одно из двух):**

1. **Запускать только один userbot для этого аккаунта.**  
   - Нужен только Nick Assistant → остановить Best:  
     `sudo systemctl stop telegram-assistant.service`  
     (при необходимости отключить автозапуск: `sudo systemctl disable telegram-assistant.service`).  
   - Нужен только Best → остановить Nick:  
     `systemctl --user stop telegram-nick-assistant.service`.

2. **Или использовать два разных Telegram-аккаунта:** Best — как сейчас; Nick Assistant — другой номер/аккаунт и отдельная сессия в каталоге Nick Assistant.

Отдельная сессия (другой файл) для Nick **без** отключения Best проблему не снимает: пока оба работают под одним аккаунтом, оба будут падать на «чужих» TL-типах в getDifference.

## Типичные ошибки по логам

Логи сервиса: `journalctl --user -u telegram-nick-assistant.service -n 100 --no-pager`.

### TypeNotFoundError (Constructor ID 94345242)

**Симптомы в логах:**
- `telethon.errors.common.TypeNotFoundError: Could not find a matching Constructor ID for the TLObject ... ID 94345242`
- Перед этим: `Cannot get difference since the account is likely misusing the session`
- Сервис падает и systemd перезапускает его (restart counter растёт).

**Причина:** Чаще всего — **одновременно запущен другой userbot на тот же аккаунт** (например Best My Assistant). Тогда в getDifference попадают TL-типы от другого клиента. См. раздел «Конфликт с Best My Assistant» выше.

**Решение:** Остановить второй userbot для этого аккаунта (или использовать другой аккаунт для Nick Assistant). Если сессия Nick была скопирована из Best — задать в `.env` новое имя сессии (`SESSION_NAME=nick_assistant_session`), удалить старый `.session` из каталога проекта и один раз войти вручную (`python main.py`).
