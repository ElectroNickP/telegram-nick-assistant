# Расследование: постоянные перезагрузки Nick Assistant и конфликт с Best My Assistant

**Дата:** 31.01.2026  
**Статус:** причина установлена, решение сформулировано.

---

## Факты по логам

### Telegram Nick Assistant (user-сервис)

- **Сервис:** `telegram-nick-assistant.service` (systemd --user)
- **Логи:** `journalctl --user -u telegram-nick-assistant.service`
- **Поведение:** стартует → логин Nick (@Pankonick) → ~15 минут работает → падение с `TypeNotFoundError: Constructor ID 94345242` → systemd перезапуск (restart counter 61, 62, 63, 64...)
- **В «Remaining bytes» ошибки:** текст от **Best My Assistant** — «Система запущена!», «Загрузка истории: X/11», «Best-ElectroNick», «june phuket», «BBD - Boat Repair», «Папка БАЗА: 11 чатов» и т.д.

### Best My Assistant (системный сервис)

- **Сервис:** `telegram-assistant.service` (systemd, не --user), **enabled**, активен
- **Логи:** `journalctl -u telegram-assistant.service`
- **Поведение:** тоже падает с `TypeNotFoundError`, но **другой** Constructor ID: **9815cec8**
- **В «Remaining bytes» ошибки Best:** текст **«Telegram Nick Assistant запущен. Мониторинг упоминаний и ответов активен.»** — то есть обновления, порождённые Nick Assistant.
- **Restart counter:** 65 и выше — Best тоже постоянно перезапускается.

## Установленные факты

| Факт | Проверка |
|------|----------|
| Один и тот же аккаунт | Оба логинятся как Nick (@Pankonick), User ID 577784602 |
| Два разных файла сессии | Nick: `.../Telegram Nick Assistant/userbot_session.session`; Best: `.../Best_my_assistant/userbot_session.session`. Файлы **разные** (cmp: differ at byte 28). |
| Оба сервиса запущены одновременно | Best — system service (active); Nick — user service (active). |
| Оба падают с TypeNotFoundError | Nick — ID 94345242; Best — ID 9815cec8. В дампе у каждого — контент от **другого** приложения. |

---

## Причина (корневая)

1. **Один Telegram-аккаунт** используется **двумя** userbot-приложениями **одновременно** (разные сессии, один аккаунт).
2. У Telegram обновления (pts/qts/seq) и поток getDifference — **на аккаунт**. В поток попадают события от обоих клиентов.
3. При вызове **getDifference()** сервер отдаёт обновления для этого аккаунта. В них могут быть TL-объекты, порождённые или связанные с **другим** клиентом (другая версия/схема Telethon, другой system_version).
4. **Nick Assistant** при разборе ответа натыкается на конструктор **0x94345242** (в потоке есть данные от Best) → Telethon не знает этот тип → TypeNotFoundError → падение → перезапуск.
5. **Best My Assistant** при разборе натыкается на конструктор **0x9815cec8** (в потоке есть данные от Nick, в т.ч. «Telegram Nick Assistant запущен...») → TypeNotFoundError → падение → перезапуск.

Итог: **конфликт не из-за одного файла сессии, а из-за одного аккаунта и двух одновременно работающих userbot’ов.** Каждый падает, когда получает в getDifference обновления с «чужим» типом TL.

---

## Решение

**Вариант A (рекомендуемый): не запускать оба userbot’а для этого аккаунта одновременно.**

- Либо **только Best**, либо **только Nick** для аккаунта Nick @Pankonick.
- Пример: если нужен только Nick Assistant — остановить Best:  
  `sudo systemctl stop telegram-assistant.service`  
  и при необходимости отключить автозапуск:  
  `sudo systemctl disable telegram-assistant.service`
- И наоборот: если нужен только Best — остановить Nick:  
  `systemctl --user stop telegram-nick-assistant.service`

**Вариант B: два разных аккаунта.**

- Best — как сейчас (аккаунт Nick @Pankonick).
- Nick Assistant — **другой** Telegram-аккаунт (второй номер, отдельный вход, отдельная сессия в каталоге Nick Assistant).
- Тогда обновления и getDifference не смешиваются между приложениями.

**Отдельная сессия (другой файл) для Nick без отключения Best — проблему не снимает:** пока оба работают под одним аккаунтом, оба будут получать в getDifference «чужие» TL-типы и падать.
